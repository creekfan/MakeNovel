# NovelAgent 面试阐述

## 项目定位

NovelAgent 是一个基于 LangChain + LangGraph 的 AI 长篇小说写作工具。

核心命题：长文本创作中，AI 如何维持角色一致性、世界观连贯性和叙事风格统一？答案是 **大纲驱动的 ReAct Agent + RAG 语义记忆 + 文风注入**。

---

## 一、人机协同意识

### 人与 AI 的分工边界

本项目的核心设计理念是"大纲由人定，正文由 AI 写"。用户设计卷→章→节三级大纲树结构，Agent 在大纲的刚性约束下自由创作。这避免了长篇小说写作中最常见的问题——AI 跑偏、人物消失、设定矛盾。

### 用 AI 工具进行开发的全流程

整个项目从概念到 28 项测试全通过的开发过程中，我深度使用了 AI 编程助手完成以下工作：

- **代码重构**：从初始的 5 Agent 管道架构重构为 LangGraph ReAct 单 Agent 架构时，AI 协助完成了 24 个文件、1787 行代码的迁移
- **缺陷检测**：通过 AI 审查发现了 ChromaDB 空集合查询崩溃、Python 3.9 不兼容 `str | None` 语法、元描述污染正文等关键 bug
- **自动化测试**：`test_all.py` 覆盖 CRUD、工具、Agent、RAG、摘要 5 大模块 28 项测试，全部由 AI 辅助生成和修复
- **极速原型验证**：从"文风学习"需求提出到完整实现（11 文件变更、API + 前端 + 存储），约 30 分钟内完成

### 人机协同的具体体现

| 协同场景 | 人的决策 | AI 的执行 |
|----------|----------|-----------|
| 架构设计 | 选定 LangGraph ReAct 而非自研循环 | 完成 create_react_agent 集成代码 |
| 元描述 bug | 分析出文风 prompt 将 LLM 带入分析模式 | 实现三层修复：System Prompt + User Message + 正则清洗 |
| 数据存储 | 选定 JSON 文件而非 SQLAlchemy（简化部署） | 完成全部 CRUD 函数和路由 |
| 文风管理 | 选定纯文本 prompt 方案而非 LLM 分析 | 实现多文风 CRUD + Agent 注入 |

## 二、全栈视野

### 技术栈

```
前端：React 18 + TypeScript + Vite + Zustand
后端：FastAPI + LangChain + LangGraph + Pydantic
AI  ：ChatOpenAI（兼容 DeepSeek/OpenAI）+ ChromaDB + sentence-transformers
存储：JSON 文件 + ChromaDB 向量持久化
通信：SSE（Server-Sent Events）流式传输
```

### 我独立完成的工作

**前端**：7 个页面

| 页面 | 关键技术点 |
|------|-----------|
| HomePage | 项目列表 CRUD |
| OutlinePage | 三级树形大纲编辑，递归节点增删改，inline 编辑 |
| EditorPage | 正文 textarea + Agent 面板 + 1s 防抖自动保存 + SSE 流式日志 + 文风下拉 |
| CharactersPage | 角色卡片管理，8 字段表单 + 关系编辑 |
| WorldPage | 世界观设定，7 分类筛选 + 内联编辑 |
| SettingsPage | LLM 参数配置，API Key Base64 编码存 localStorage |
| StylePage | 多文风管理，左侧列表 + 右侧编辑器 |

**Agent 面板实现要点**：

- 不使用 `EventSource`（只支持 GET），改用 `fetch` + `ReadableStream` + `TextDecoder` 手动解析 SSE
- Buffer 机制处理 TCP 分包：解码后按 `\n` 分割，`data:` 前缀匹配，残留数据留在 buffer
- 实时推送 `init → thinking → tool_call → complete` 事件，前端按颜色/图标分类渲染

**后端**：6 个路由模块，2 个 Agent 端点

```
/api/novels     → CRUD
/api/outlines    → 大纲 + 节正文 CRUD
/api/characters  → 角色 CRUD
/api/world       → 世界观 CRUD
/api/styles      → 文风 CRUD
/api/agent/run   → LangGraph ReAct Agent（SSE 流式）
/api/agent/summrize → 独立摘要生成
```

### Chatbot 交互界面

EditorPage 的 Agent 面板本质上就是一个 Chatbot 界面：
- 自然语言指令输入
- Agent 实时推理日志
- 工具调用过程可视化
- 最终结果自动写入编辑器

符合面试要求中"能独立完成 Chatbot 交互界面开发"的标准。

## 三、底层优化

### 当前项目的底层技术

| 技术 | 选型理由 |
|------|----------|
| LangGraph `create_react_agent` | 替代自研 ReAct 循环，标准化工具调用 + 状态管理 |
| ChromaDB PersistentClient | 本地持久化向量数据库，零配置，单文件部署 |
| sentence-transformers `all-MiniLM-L6-v2` | 384 维轻量嵌入，CPU 运行，无需外部 Embedding API |
| `astream_events` (v2) | LangGraph 原生事件流，替代手动事件发布，减少样板代码 |

### 优化的具体实践

1. **嵌入模型选择**：对比了 OpenAI text-embedding-3-small（需 API 调用、收费）和 all-MiniLM-L6-v2（本地免费、毫秒级延迟），选择了后者——对写作辅助场景，384 维语义相似度足够区分不同章节

2. **Agent 输出清洗**：实现了 `_clean_content()` 正则清洗函数，处理 LLM 输出中的元描述污染问题。当 LLM 输出 `"正文已创作完成。这一节围绕XX展开，运用了YY技术……"` 时，自动剥离元描述，保留纯正文

3. **内容保存阈值**：`embed_section()` 设置 50 字最小阈值，避免空白或极短内容污染向量空间

4. **ChromaDB 空集合处理**：`search_sections()` 在查询前检查集合是否为空，避免 ChromaDB 空查询异常

### 可扩展的底层优化方向

| 方向 | 方案 | 价值 |
|------|------|------|
| 微调 | LoRA 微调 Qwen2.5-14B，注入特定文风训练数据 | 产出真正"像作者"的文风，而非 prompt 模拟 |
| 本地部署 | Ollama + Qwen2.5 替代 DeepSeek API | 降低 API 成本，数据不出本地 |
| 推理加速 | vLLM 部署，优化长文本生成的 token 吞吐 | 长章节生成从 60s → 15s |
| 嵌入升级 | BGE-M3 替代 all-MiniLM-L6-v2 | 多语言 + 8192 token 长文本嵌入 |

## 四、产品触觉

### 我对 AI Agent 赛道的理解

AI 长篇小说写作是一个**强需求但无成熟方案**的赛道。现有产品的问题：

| 产品 | 核心问题 |
|------|----------|
| GPT/Claude 裸写 | 窗口限制，5 万字后完全失忆，角色混乱 |
| NovelAI | 偏日轻风格，对中文严肃文学支持弱 |
| 彩云小梦 | 用户不可控，"一键生成"缺乏创作参与感 |

### NovelAgent 的产品设计理念

**1. 大纲驱动 → 人控制结构，AI 填充内容**
- 与传统"一键生成"不同，用户拥有创作主导权
- 三级大纲树（卷/章/节）给 AI 刚性边界，避免跑题

**2. RAG 记忆 → 解决长文本遗忘**
- ChromaDB 持久化嵌入，不限 token 窗口
- Agent 通过 `search_memory` 主动检索前文，维持人物一致性

**3. 文风注入 → 不靠微调实现风格统一**
- 用户自行搜集写作技法 prompt（如"江南的三种核心技术"）
- Agent 创作时自动前置注入，比微调更灵活、可迭代
- 多文风管理：不同小说可用不同风格

**4. 对行业爆款的借鉴与反思**

| 产品 | 借鉴 | 反思 |
|------|------|------|
| Cursor | AI 辅助而非替代，人保持主控权 | 写作工具同理，大纲驱动是正确方向 |
| Perplexity | 引用来源、可追溯 | NovelAgent 的 search_memory 可展示检索到的前文 |
| ChatGPT Canvas | 并排编辑 + AI 协作 | EditorPage 的左侧编辑器 + 右侧 Agent = 同款模式 |

### 产品的独特见解

- **"文风"不应是 prompt 模板**：我选择让用户自己搜集、管理文风 prompt（而非黑盒分析），因为真正的写作风格需要人的审美判断，机器无法替代
- **长文本 Agent 的核心是记忆，不是推理**：对于小说创作，ChromaDB RAG 的检索质量远比 Agent 的 CoT/ReAct 复杂度更重要
- **SSE 流式是 Agent 交互的及格线**：用户需要看到 Agent 每一步在做什么（思考→查资料→构思→写作），透明化建立信任
