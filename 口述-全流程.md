# 口述脚本 — NovelAgent 项目全流程

> 闭眼练习，5-8 分钟内流畅讲完。

---

## 一、项目概览（30 秒）

NovelAgent 是一个基于 LangChain + LangGraph 的 AI 长篇小说写作工具。核心命题：长文本创作中，AI 如何维持角色一致性、世界观连贯性和叙事风格统一？

解法：**大纲驱动的 ReAct Agent + ChromaDB RAG 语义记忆 + 文风注入**。

技术栈：React + TypeScript + Vite（前端），FastAPI + LangChain + ChromaDB（后端）。

---

## 二、用户操作全流程（2 分钟）

### 2.1 创建项目

用户在首页输入小说名，POST `/api/novels` 创建项目。后端在 `data/{novel_id}/` 下初始化 JSON 文件目录——meta.json、outline.json、characters.json、world_settings.json、summaries.json 各就各位。

### 2.2 搭建大纲

进入大纲页，设计卷→章→节三级树形结构。每个节点有标题、情节概要、状态标记。树通过递归组件渲染，支持内联增删改。保存时 PUT `/api/novels/{id}/outline`，写 outline.json。

大纲是人设计的边界，Agent 不修改大纲。这是"人机协同"的核心——人控制结构，AI 填充内容。

### 2.3 填写角色和世界观

角色卡片页填写 8 个字段：名称、定位、外貌、性格、背景、能力、说话风格、弧光、关系网。世界观设定页按 7 类分类（环境、势力、规则、种族、物品、职业、历史）管理。

数据全部通过 REST API 持久化到 JSON 文件。

### 2.4 管理文风

侧边栏进入「文风」页面。用户可以搜集保存多个文风 prompt——比如"江南的三种核心技术"、"古龙式短句风格"。每个文风是一个纯文本 markdown 文件，存在 `data/{novel_id}/styles/{id}.md`。

### 2.5 进入写作

从大纲点击某节进入编辑器。编辑器是 textarea + Agent 面板的双栏布局。

Agent 面板从上到下：创作指令输入框、文风下拉选择器、「运行 Agent」按钮、「生成摘要」按钮、执行日志区域。

## 三、Agent 执行全链路（2 分钟）

### 3.1 前端发起 SSE 请求

用户点击「运行 Agent」触发 `runAgent()`。函数做三件事：

第一，从 Zustand store 取 LLM 配置（api_key/model/base_url/temperature/max_tokens），API Key 为空则弹窗提醒。Key 是 Base64 编码存在浏览器 localStorage 的。

第二，1 秒防抖 timer 自动把编辑器已有内容 POST 到后端保存。

第三，调用 `runAgentStream()`。用 fetch POST 到 `/api/novels/{id}/agent/run`，body 携带 section_id、instruction、style_id 和全部 LLM 参数。

为什么不用 EventSource？EventSource 只支持 GET，而我们必须 POST 传 API Key 和大量参数。

请求发出后，前端拿到 `res.body`，调用 `getReader()` 获取 ReadableStream。进入 `while(true)` 循环，每次 `await reader.read()` 拿字节数据，TextDecoder 解码后追加到 buffer。buffer 按换行 `\n` 分割，匹配 `data:` 前缀的行，JSON.parse 解析后调用 `onEvent` 回调更新 React 状态。

buffer 机制很重要——TCP 分包可能导致不完整的 SSE 行，残留数据留在 buffer 等待下一轮拼接。

### 3.2 后端构建上下文

请求到达 FastAPI，路由 `run_agent` 用 Pydantic 校验参数，实例化 NovelAgent。

NovelAgent 构造函数：
```python
self.agent = create_react_agent(
    model=ChatOpenAI(api_key, base_url, model),
    tools=get_all_tools(),
    prompt=AGENT_SYSTEM_PROMPT,
)
```

`astream()` 方法开始构建 user message。步骤如下：

1. 读取大纲 JSON → `_find_sections()` 递归遍历卷→章→节树，找到当前节的标题、概要、状态
2. 检查是否传了 `style_id`。如果有，从 `data/{novel_id}/styles/{id}.md` 读文风内容，前置到 user message：`## 文风要求（作为写作技法指南，在创作中自然运用，不要描述你如何运用的）{文风正文}`
3. 拼接任务指令、节信息、已有内容（前 1000 字）
4. 末尾加强指令：`重要：finish 的参数必须只包含小说正文本身，不要包含任何创作总结、技术分析或过程描述。`

### 3.3 Agent ReAct 循环

`agent.astream_events()` 启动事件流，SSE 推送给前端。

Agent 的执行流程：

```
用户指令 → LLM 思考 → 工具调用 → 观察结果 → LLM 思考 → ... → finish
```

**6 个工具**，全部基于 `BaseTool` + Pydantic `args_schema`：

| 工具 | 数据源 | 做什么 |
|------|--------|--------|
| `get_outline` | outline.json | 递归遍历大纲树，输出卷→章→节结构 |
| `get_characters` | characters.json | 格式化角色卡片（外貌、性格、背景、关系） |
| `get_world_settings` | world_settings.json | 按类别输出世界观设定 |
| `get_summaries` | summaries.json | 读取最近 5 条已完成节摘要 |
| `search_memory` | ChromaDB | 用 sentence-transformers 编码查询，语义搜索前文 |
| `finish` | — | 提交最终正文，触发循环终止 |

**事件监听（4 类）**：

1. `on_chat_model_start` → 前端显示 "思考中..."
2. `on_chat_model_end` → 前端显示 "思考完成"。如果 LLM 返回无 tool_calls 的文本，捕获为 `final_content`
3. `on_tool_start` → 前端显示 "调用工具：get_outline"，含输入参数前 200 字
4. `on_tool_end` → 前端显示工具执行完成。如果工具名是 `finish`，从 tool_output 提取 `final_content`

### 3.4 内容提取与清洗

`final_content` 的来源有两个优先级：

第一优先级：`finish` 工具的 output。Agent 创作完正文后调用 `finish(result=正文)`，我们从 `on_tool_end` 事件中捕获。

第二优先级（fallback）：如果走到循环结束还没拿到 `final_content`，重新遍历最终状态的 messages 列表，从最后一条不带 tool_calls 的 AIMessage 中提取。

拿到 `final_content` 后，经过 `_clean_content()` 正则清洗：

```python
_META_PATTERNS = [
    r"^正文已创作完成[。，]?",       # 匹配"正文已创作完成。"
    r"^本节(写的是|围绕|讲述).*?[。，]", # 匹配"本节围绕XX展开"
    r"运用了.*?(技术|手法)",           # 匹配"运用了XX技术"
    r"完整呈现了.*?[。]",             # 匹配"完整呈现了XX转折"
    ...
]
```

7 种模式逐条正则替换。洗完后检查：如果原文 ≥ 50 字但清洗后 < 50 字，说明误删了正文，回退到原文。

这解决了元描述污染的 bug——文风 prompt 的技法描述格式会把 LLM 带入分析模式，输出"正文已创作完成。这一节运用了XX技术……"而非纯正文。

### 3.5 保存与 RAG 嵌入

清洗后的 `final_content` 两个去向：

1. 文件存储：`storage.save_section_content()` → `sections/{sectionId}.txt`
2. 向量嵌入：`memory.embed_section()`。内容 > 50 字时，用 `sentence-transformers/all-MiniLM-L6-v2` 编码为 384 维向量，upsert 到 ChromaDB 的 `novel_{novel_id}` 集合

ChromaDB 使用 `PersistentClient`——数据存在 `data/chroma/` 目录，零运维，一个文件夹就是一个数据库。

## 四、摘要生成（30 秒）

摘要是独立端点 `/api/novels/{id}/agent/summrize`，不走 Agent 循环。

流程：用 `ChatOpenAI` 直接调用，system prompt 加载 `skills/summary.md`，user 传入正文。LLM 返回 JSON `{summary, key_events, character_state_changes, world_setting_changes}`。

解析后更新大纲节点（summary + status → "done"），追加到 summaries.json。后续 Agent 创作时通过 `get_summaries` 工具自动引用这些摘要，形成跨章节的上下文链。

## 五、数据架构（30 秒）

```
backend/data/{novel_id}/
├── meta.json              # 项目元信息（id、name）
├── outline.json           # 大纲树（volumes → chapters → sections）
├── characters.json        # 角色卡片列表
├── world_settings.json    # 世界观设定列表
├── summaries.json         # 已完成节摘要列表
├── styles/
│   ├── list.json          # [{id, name, created_at}]
│   └── sty_xxx.md         # 文风 prompt 正文
├── sections/
│   └── {section_id}.txt   # 各节正文
└── ../../chroma/          # ChromaDB 持久化目录
    └── novel_{id}/        # 每部小说独立集合
```

选 JSON 而非 SQLite 的原因：零迁移、可读可追溯、每部小说独立目录便于备份。

## 六、核心设计决策（1 分钟）

### 6.1 为什么是 ReAct Agent 而非多 Agent 管道？

V1 是 5 Agent 管道（Preparer → Creator → Reviewer → Reviser → Polisher）。问题：固定流程不灵活、工具是 placeholder、无跨章节记忆。V2 改用 LangGraph ReAct——流程由 LLM 动态决策，6 个工具都有真实的数据读取逻辑，ChromaDB 提供持久化记忆。

### 6.2 为什么文风用纯文本 prompt 而非 LLM 分析？

用户搜集的文风 prompt 质量远高于机器分析——人类作家知道什么是好技法、什么是反例。LLM 自动分析只能输出模板化的"句式较长/用词文学化"，缺乏可操作的写作指令。

### 6.3 为什么工具里没有"写作"工具？

写作逻辑放在 LLM 推理中而非工具中——减少工具数量、降低 Function Calling 选择错误率。Agent 通过 `finish` 工具提交结果，真正的写作发生在 ReAct 循环的思考阶段。

### 6.4 元描述 bug 是怎么解决的？

不是单点 fix，而是三层防御：System Prompt 把禁止规则提到「最高法则」开头、User Message 强化输出指令、`_clean_content()` 正则清洗兜底。根因是文风 prompt 的格式先验——LLM 从技法描述格式中 learn 到的不是"用这个技法写"而是"用描述技法的格式输出"。
